Кейс. «Помощник выбора на полке» на основе данных «Парфюмерного органа» и каталога ароматов

Партнёр — SensoryLAB — российская компания, специализирующаяся на нейротехнологиях и сенсорных устройствах. Один из продуктов — интерактивный стенд «Парфюмерный орган»: пользователь нажимает каналы/ароматы, регулирует интенсивность и может сохранить «рецепт» (смесь интенсивностей). Устройство фиксирует события взаимодействия без персональных данных, либо картами лояльности.

Нужно разработать прототип системы, которая по текущему поведению пользователя на «Парфюмерном органе» и/или по его сохранённому рецепту в реальном времени рекомендует конкретные SKU из каталога, которые лучше всего соответствуют выбранной «смеси» и нотам.


1) Что такое «Парфюмерный орган»
Физическое интерактивное устройство, которое:
- имеет несколько каналов (например, 6/8/12), соответствующих базовым ароматическим компонентам/аккордам;
- позволяет нажимать каналы, менять уровни интенсивности;
- позволяет сохранять «рецепт» смеси (пример формата: `0:49,1:80,2:50,3:40,4:63,5:50`);
- логирует действия пользователя в рамках сессии.

Это источник поведенческих данных (может быть привязан к картам лояльности)


2) Бизнес-цель проекта
Сделать «помощник выбора на полке»: когда пользователь взаимодействует с органом и собирает смесь, система должна:
- быстро (онлайн) понять предпочтения пользователя по нотам/профилю;
- выдать Top-N SKU из ассортимента (каталога) — «что взять с полки прямо сейчас»;
- (опционально) объяснить рекомендации: какие ноты/компоненты дали вклад;
- (опционально) учитывать обратную связь (просмотр/выбор/покупка) для улучшения качества.

Ключевой акцент: рекомендация определяется не одной нотой, а смесью (рецептом) + динамикой нажатий.


3) Пользовательский сценарий (MVP)
1) Пользователь нажимает каналы и/или сохраняет рецепт смеси (интенсивности по каналам).
2) Система строит профиль предпочтений пользователя (на основе смеси и/или логов нажатий).
3) По кнопке «Подобрать на полке» (или через API) система возвращает Top‑N SKU.
4) (Опционально) фиксируется обратная связь: что пользователь выбрал/купил.


4) Источники данных (Lite выдаётся всем)
В кейсе используются два набора данных в формате Parquet/CSV, подготовленные скриптами (в проекте они уже генерируют README/MD с описанием колонок):

A) Каталог ароматов / SKU (Lite)
Папка: `student_dataset/lite/`
Ключевые таблицы:
- `perfumes.parquet` — справочник SKU (бренд, название, численные признаки популярности/характеристик)
- `perfume_notes.parquet` — ноты SKU с весами `votes`
- `perfume_notes_agg.parquet` — агрегированные ноты (списком) для быстрого прототипирования
- (опционально) `comments_agg.parquet` — только количество комментариев (без текста)
Основной ключ: `perfume_id` (int)

B) Логи «Парфюмерного органа» (Lite, синтетический набор для воспроизводимых метрик)
Папка: `organ_dataset_synth/lite/`
Ключевые таблицы:
- `organ_sessions.parquet` — сессии (включая `target_perfume_id` как ground-truth для оценки)
- `organ_presses.parquet` — события нажатий (channel/aroma, длительность, интенсивность)
- `organ_recipes.parquet` — сохранённый рецепт строкой (формат `0:49,1:80,...`)
- `organ_recipe_components.parquet` — рецепт в разрезе каналов (channel_index, intensity)
- `organ_aroma_notes_map.parquet` — маппинг `aroma_id -> note` (как переводить орган в пространство нот)
- `organ_aromas.parquet` — справочник аром/каналов (человекочитаемо)
- `organ_feedback.parquet` — синтетическая обратная связь (view/purchased)
Основные ключи:
- `session_id` — ключ сессии
- `target_perfume_id` — целевой SKU для метрик, джойнится на `perfumes.perfume_id`

Важно: Lite-орган — синтетика, чтобы:
- не передавать реальные логи публично,
- иметь “правильную” разметку для оценки качества рекомендаций (Hit@K/MRR/NDCG).

Полный (Full) набор данных
- Полные версии каталога/логов (если применимо) предоставляются по запросу, после простого NDA.
- Требование к решению: оно должно работать на Lite и масштабироваться на Full без переписывания логики (только замена входных файлов/параметров).


5) Цель проекта (что нужно сделать)
Разработать прототип рекомендательной системы, которая:
1) Загружает и обрабатывает Lite-датасеты.
2) Строит профиль предпочтений пользователя:
   - по смеси (рецепту): интенсивности по каналам;
   - и/или по логам нажатий: учитывая длительность, интенсивность, давность событий.
3) Ранжирует каталог SKU и выдаёт Top‑N рекомендаций.
4) Считает метрики качества на тестовой выборке (по `target_perfume_id`).
5) Сравнивает результат с бейзлайнами.
6) Предоставляет демонстрацию инференса (API или CLI), чтобы можно было “проверить руками”.


6) Архитектура решения (рекомендовано)
Backend: Python
Формат демо: REST API и/или CLI (дашборд и личный кабинет не требуются)
Хранение данных: Parquet/CSV (Lite), возможность переключиться на Full
Ограничение: решение должно запускаться локально (8–16 GB RAM) и выдавать рекомендацию быстро (ориентир: сотни миллисекунд на сессию при Lite).


7) Основная DS/ML задача (обязательная)
A. Рекомендация SKU по текущей сессии/рецепту (Ranking / Retrieval)
Вход:
- `session_id` (и данные этой сессии из `organ_*`)
- рецепт (как `organ_recipe_components`, либо парсинг строки recipe)
- (опционально) события нажатий `organ_presses`

Выход:
- Top‑N SKU (`perfume_id`) с score
- (желательно) объяснение: вклад нот/компонентов в score

Требование: скоринг должен учитывать “смесь”, а не только “одну лучшую ноту”.
Пример логики профиля:
- user_vector(note) = сумма вкладов нот из каналов с весом intensity/100 (и весом маппинга aroma->note)
- sku_vector(note) = log1p(votes) или другая нормализация
- score = cosine(user_vector, sku_vector) или нормализованный dot-product
- (опционально) комбинировать рецепт и нажатия: u_total = α*u_recipe + (1-α)*u_presses


8) Дополнительные задачи (на выбор, для усиления решения)
B. Аналитика “что в топе по SKU” (без дашборда, как отчёт/скрипт)
Сделать скрипт/ноутбук, который отвечает:
- какие SKU чаще всего попадают в Top‑5/Top‑10 по всем сессиям;
- какие ноты доминируют в рецептах;
- какие каналы/аромы чаще всего приводят к каким SKU (таблица соответствий).

C. Улучшение качества ранжирования
- reranking кандидатов на синтетической разметке (view/purchased/target),
- учет diversity в выдаче,
- учет стабильности выдачи при небольших изменениях рецепта.

D. Оптимизация набора каналов органа (6–12 базовых компонент)
По каталогу SKU и нотам предложить:
- как выбрать 6–12 базовых направлений/нот, чтобы лучше покрывать ассортимент,
- обосновать метод (кластеризация, покрытие множеств и т.п.),
- показать метрики покрытия/различимости.


9) Метрики и KPI
Для основной задачи рекомендаций (на Lite синтетике):
- Hit@K (K=5/10)
- MRR
- NDCG@K

Дополнительно (по желанию):
- Coverage (сколько разных SKU система вообще рекомендует)
- Diversity (насколько разнообразна выдача Top‑N)
- Latency (время ответа сервиса)


10) Бейзлайны (обязательное сравнение)
Нужно реализовать минимум 2–3 бейзлайна и сравнить с ними:
1) Top Popular SKU: по `allVotes` (или clslove/clslike)
2) Простое совпадение по нотам: weighted overlap / Jaccard (по нотам без учета последовательности)
3) Single-signal baseline: только самая интенсивная нота/канал → рекомендации по SKU с этой нотой

Требование: в отчёте должно быть сравнение метрик модели и бейзлайнов на одном сплите.

14) Критерии оценивания (максимум 60 баллов)

1) Соответствие требованиям брифа (10 баллов)
10 — есть работающий прототип рекомендаций Top‑N по session/recipe, можно проверить
5 — частично (например, только ноутбук без демо-инференса)
0 — проверить невозможно

2) Ценность для сценария “поиск на полке” (10 баллов)
10 — рекомендации релевантны смеси, есть объяснение (ноты/вклад), понятный контракт вход/выход
5 — рекомендации есть, но слабо объяснимы/не учитывают смесь
0 — не решает задачу рекомендации SKU

3) DS/ML качество и сравнение с бейзлайнами (15 баллов)
15 — метрики Hit@K/MRR/NDCG@K, корректный сплит, сравнение минимум с 2–3 бейзлайнами, интерпретация
7 — модель есть, но сравнение/метрики неполные
0 — DS/ML часть отсутствует или некорректна

4) Корректная работа с данными и маппингом “орган → ноты → SKU” (10 баллов)
10 — прозрачный пайплайн, корректные джойны по ключам, аккуратная обработка нот, рецепт парсится и используется
5 — много допущений без проверки, слабая работа с ключами/нотам
0 — связи данных не подтверждены, результат случайный

5) Инженерная реализация и real-time готовность (10 баллов)
10 — быстрый инференс, стабильная выдача, понятный API/CLI, контроль версий модели/параметров
5 — работает, но медленно/нестабильно/непонятный интерфейс
0 — демо-инференса нет

6) Документация и воспроизводимость (5 баллов)
5 — Docker + README + фиксированные зависимости, запуск “одной командой”
2 — частично
0 — не воспроизводится


Ожидаемый результат
Прототип рекомендательной системы, которая по данным «Парфюмерного органа» (смесь рецепта + нажатия) рекомендует релевантные SKU из каталога и измеримо превосходит бейзлайны на Lite-датасете.
